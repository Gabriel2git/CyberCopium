# 赛博吸氧机 / Cyber Copium - MVP 开发计划

## 一、项目概述

**产品定位**: 面向年轻用户的 AI 情绪转译与微行动工具  
**产品形态**: Web 单页应用  
**一句话定位**: 当现实太窒息时，给用户一口"可转发、可执行、带一点荒诞感"的赛博精神氧气。  
**Slogan**: 现实太窒息？来吸一口纯度 100% 的赛博盲目乐观。

### 核心闭环
用户输入困境 → 状态识别 → 生成五件套 (状态命名/抽象嘴替/贴纸文案/荒诞动机/最小行动) → 用户反馈

---

## 二、技术栈

- **前端**: Next.js + TypeScript + Tailwind + shadcn/ui
- **后端**: Next.js Route Handlers
- **数据库**: Supabase / PostgreSQL
- **向量检索**: pgvector
- **模型**: OpenAI API (生成 + embedding)
- **贴纸渲染**: HTML/CSS/Canvas 或 Satori + Resvg
- **分析埋点**: PostHog
- **部署**: Vercel

---

### 阶段二：真实生成闭环 (Epic 2 - AI 生成链路)

**目标**: 从假产品变成真产品，接入真实 AI 生成能力

#### 任务 2.1: API 接口层
- [ ] 创建 `/api/generate` Route Handler
- [ ] 定义请求/响应数据结构
- [ ] 实现 OpenAI API 基础调用
- [ ] 实现错误处理与超时处理
- [ ] 返回结构化 JSON

#### 任务 2.2: Prompt v1 设计
- [ ] 设计五件套输出字段结构:
  - `status_label`: 状态命名 (≤20 字)
  - `voice_line`: 抽象嘴替 (≤50 字)
  - `sticker_text`: 贴纸文案 (≤30 字)
  - `absurd_motivation`: 荒诞动机 (≤40 字)
  - `micro_action`: 最小行动 (≤30 字，3 分钟内可执行)
  - `tone_level`: 语气强度 (L1/L2/L3)
- [ ] 编写 Prompt 模板
- [ ] 明确字数要求、语气要求、禁止事项
- [ ] 测试 Prompt 稳定性

#### 任务 2.3: 状态识别模块
- [ ] 定义状态标签集:
  - 拖延/回避
  - 羞耻/自我否定
  - 疲惫/低启动
  - 轻度发疯/情绪过载
  - 想做但卡住
- [ ] 定义语气强度三挡:
  - L1 稳定模式：低梗、低刺激、偏现实支持
  - L2 轻抽象模式：轻微自嘲、温和荒诞
  - L3 高抽象模式：允许更明显的发疯文学与抽象嘴替
- [ ] 实现状态识别逻辑 (基于 AI 或规则)
- [ ] 实现语气强度映射逻辑

#### 任务 2.4: 微行动桥接
- [ ] 在 Prompt 中强化"荒诞动机"到"最小行动"的自然过渡
- [ ] 确保最小行动具体、可执行、低门槛
- [ ] 测试不同场景下的行动建议质量

#### 任务 2.5: 错误兜底机制
- [ ] 处理生成失败 (模型超时、API 错误)
- [ ] 处理字段缺失 (使用 fallback 模板)
- [ ] 处理文案超长 (后端截断 + 前端降级)
- [ ] 实现友好错误提示
- [ ] 提供"重新生成"入口

**阶段二验收标准**:
- 用户真实输入后，能收到 AI 生成的完整五件套
- 字段齐全、结构稳定
- 异常情况有友好兜底
- 不同状态输入得到不同强度输出

## 四、开发顺序建议

### 推荐顺序
1. **阶段一 (假数据闭环)**: 1-2 天
2. **阶段二 (真实生成闭环)**: 2-3 天
3. **阶段三 (贴纸与视觉稳定性)**: 1-2 天
4. **阶段四 (语料与风格控制)**: 2-3 天
5. **阶段五 (反馈与演示准备)**: 1 天

**总计**: 7-11 天完成 MVP

### 开发原则
- 先做单链路闭环，再补边角功能
- 先保证结果可用，再追求风格惊艳
- 先做静态模板和规则兜底，再接 AI 动态生成
- 先做主路径，不在历史记录、账号、分享链路上分心

## 六、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| OpenAI API 不稳定 | 生成失败 | 实现重试机制 + 本地 fallback 模板 |
| 贴纸文案超长 | 排版炸版 | 后端长度限制 + 前端自适应缩放 |
| 风格不稳定 | 输出忽高忽低 | 语料库检索 + 强度控制 |
| 开发周期过长 | 无法按时演示 | 优先保证 P0 功能，砍掉非必要功能 |
