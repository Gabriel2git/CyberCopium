# 单层架构可行性探讨

## 当前双层架构回顾

### 第一层：Analyzer（分析层）
**职责**：
- 分析用户输入
- 提取结构化数据：scene, status_tag, tone_level, style_mode, risk_level, action_window, primary_block
- 风控：识别高风险内容并降级

**输出示例**：
```json
{
  "scene": "work",
  "status_tag": "拖延/回避",
  "tone_level": "L2",
  "style_mode": "体面发疯",
  "risk_level": "low",
  "action_window": "3min"
}
```

### 第二层：Composer（生成层）
**职责**：
- 基于分析结果生成内容
- 五件套输出：status_label, voice_line, sticker_text, absurd_motivation, micro_action
- 遵循风格约束和长度限制

**输出示例**：
```json
{
  "status_label": "高优级拖延症发作中",
  "voice_line": "当代年轻人的日常：在逃避中寻找自我",
  "sticker_text": "禁止蕉绿",
  "absurd_motivation": "研究表明，拖延是人类进化出的高级技能",
  "micro_action": "打开电脑，双击任何一个图标"
}
```

## 单层架构方案

### 方案 A：合并 Prompt（推荐）

将两层 Prompt 合并为一个，在一次调用中同时完成分析和生成：

```typescript
const SINGLE_LAYER_PROMPT = `
# 角色
你是赛博吸氧机，一次性完成分析和生成。

# 分析任务
分析用户输入，提取以下结构化数据：
- scene: 场景 (job_search/study/work/relationship/daily_life)
- status_tag: 状态标签 (拖延/回避|羞耻/自我否定|...)
- tone_level: 语气 (L1|L2|L3)
- style_mode: 风格模式
- risk_level: 风险等级
- action_window: 行动时间窗口

# 生成任务
基于分析结果，生成五件套内容...

# 约束
1. 必须先分析再生成
2. 严格遵循 JSON 格式
3. ...
`;
```

**输出格式**：
```json
{
  "analysis": {
    "scene": "work",
    "status_tag": "拖延/回避",
    "tone_level": "L2",
    ...
  },
  "result": {
    "status_label": "...",
    "voice_line": "...",
    ...
  }
}
```

### 方案 B：跳过分析，直接生成

完全省略分析步骤，根据输入直接生成内容：

**优点**：最快（只有一次调用）
**缺点**：难以控制风格一致性和风险

---

## 单层架构的优势

| 方面 | 说明 |
|------|------|
| **延迟** | 减少 50%（去掉一次 API 调用） |
| **成本** | 减少 50%（API 调用减半） |
| **代码复杂度** | 更简单，更易维护 |
| **用户体验** | 响应更快 |

---

## 单层架构的风险

### 1. 质量控制难度增加

**问题**：模型需要同时完成分析和生成，可能在某一方面妥协

**影响**：
- 风格可能不够一致
- 长度可能超标
- 格式可能不标准

### 2. 风控能力下降

**问题**：没有独立的分析层，风险识别可能不够准确

**影响**：
- 高风险内容可能漏检
- L1 降级可能不执行

### 3. 调试困难

**问题**：没有中间结果可参考

**影响**：
- 出问题时难以定位原因
- 无法单独分析用户输入特点

### 4. Prompt 工程复杂度

**问题**：单层 Prompt 需要同时描述所有约束

**影响**：
- Prompt 更长更复杂
- 模型可能忽略部分约束

## 关键问题探讨

### Q1: 单层能否保持风格一致性？

**可能方案**：
- 使用更强的模型（如 qwen-plus）
- 更详细的 Prompt 约束
- 添加输出验证逻辑

**风险**：需要测试验证

### Q2: 风控如何保证？

**可能方案**：
- 在单层 Prompt 中强化风控指令
- 添加后置验证：检查输出中的风险关键词
- 敏感词过滤

**风险**：没有独立的分析层准确率高

### Q3: 输出格式如何保证？

**可能方案**：
- JSON Schema 验证
- 重试机制：格式不对则重新生成
- 后处理：强制修正格式

**风险**：增加延迟和成本

---

## 建议的验证步骤

如果决定尝试单层架构：

1. **Prompt 设计**：设计合并后的单层 Prompt
2. **本地测试**：手动测试 20-30 个case
3. **质量对比**：对比双层和单层的输出质量
4. **A/B 测试**：线上小流量测试
5. **指标监控**：监控风格一致率、格式正确率

---

## 结论

**可以尝试单层架构**，但需要注意：

1. ⚠️ 质量可能略有下降
2. ⚠️ 需要更强的模型或更复杂的 Prompt
3. ⚠️ 建议先小流量测试
4. ✅ 延迟和成本会显著降低

**建议**：保持双层架构作为备选，在单层效果不佳时回退。
