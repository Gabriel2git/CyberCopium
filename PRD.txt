精简版 PRD｜赛博吸氧机 / Cyber Copium
一、基础信息
产品名称： 赛博吸氧机 / Cyber Copium
产品类型： 面向年轻用户的 AI 情绪转译与微行动工具
产品阶段： MVP
产品形态： Web 单页应用
一句话定位： 当现实太窒息时，给用户一口“可转发、可执行、带一点荒诞感”的赛博精神氧气。
Slogan： 现实太窒息？来吸一口纯度 100% 的赛博盲目乐观。
________________________________________
二、立项目标
当前年轻用户在面对求职、学习、工作和关系压力时，越来越依赖抽象表情包、发疯文学、自嘲嘴替来完成情绪缓冲。但现有鼓励类产品存在两个明显问题：表达过时，像模板鸡汤；能安慰但不能推动行动。
本项目目标不是做一个泛心理陪伴产品，而是做一个高完成度、小闭环的 AI 工具：用户输入一句“我现在卡在哪”，系统输出一组符合年轻人语境的结果，帮助用户从“闷住”切换到“先动一下”。
________________________________________
三、目标用户
核心用户： 18–30 岁，熟悉网络语境的学生和年轻职场人。
共性特征： - 常在任务压力下进入拖延、羞耻、轻度崩溃状态 - 对传统励志表达不耐受 - 更愿意接受带抽象感、自嘲感、荒诞感的内容 - 有截图、转发、保存“嘴替文案”和表情包的习惯
非目标用户： - 寻求专业心理治疗或危机支持的用户 - 对网络语言和抽象表达明显排斥的用户
________________________________________
四、核心问题
用户并不缺道理，缺的是一种能命中当下语境的表达方式，以及一个足够小、足够轻、能够立即开始的动作。
具体表现为： - 明明知道该做什么，但就是动不了 - 很难准确表达自己现在到底怎么了 - 被空泛安慰激怒，觉得“不懂我” - 即使被逗笑，也无法自然转化成行动
同时，产品在落地时存在四个关键卡点，需要在 PRD 层提前收口： - 微行动如果衔接生硬，会破坏前半段建立起来的荒诞语境 - 语料库如果完全手工冷启动，时间成本过高，容易拖慢 MVP 进度 - 贴纸文案如果长度不可控，前端渲染容易溢出，影响完成度 - 抽象语气如果没有强度控制，容易在不合适场景下“疯过头”
________________________________________
五、产品方案
产品围绕一个极简闭环展开：
用户输入一句当前困境 → 系统识别状态与语气强度 → 生成抽象嘴替与精神贴纸 → 通过“荒诞动机”桥接到最小行动 → 用户反馈是否命中、是否想重来、是否开始行动。
产品不追求长对话陪伴，而是追求一次响应内的高命中率和高压缩表达。
________________________________________
六、核心价值
价值一：语境命中。
让用户感觉“这话像我会说的”，而不是“AI 在营业”。
价值二：情绪转译。
把混乱、自责、回避翻译成可以被观看、被转发、被消化的内容。
价值三：行动收束。
每次输出不止停在内容上，而是通过“荒诞动机”平滑过渡到一个 3 分钟内可执行的动作。
价值四：风格可控。
不是一味发疯，而是根据场景动态控制抽象强度，保证表达既有味道又不过界。
________________________________________
七、MVP 范围
7.1 本期要做
1. 输入模块
支持用户输入一句自然语言，描述自己当前卡住的事。
2. 状态识别模块
基于输入，识别用户当前所处状态，输出结构化标签，例如： - 拖延 / 回避 - 羞耻 / 自我否定 - 疲惫 / 低启动 - 轻度发疯 / 情绪过载 - 想做但卡住
同时输出语气强度建议，用于控制结果的抽象程度，默认分为： - L1 稳定模式：低梗、低刺激、偏现实支持 - L2 轻抽象模式：轻微自嘲、温和荒诞 - L3 高抽象模式：允许更明显的发疯文学与抽象嘴替
3. 语境生成模块
结合状态标签、语气强度与语料模板，输出以下五件套： - 状态命名 - 抽象嘴替 - 贴纸文案 - 荒诞动机 - 最小行动
4. 贴纸生成模块
基于固定模板与动态文案，生成一张可保存的精神贴纸。
5. 反馈模块
用户可对结果做最基础反馈： - 像不像我想说的 - 有没有帮我动起来 - 重新生成 - 我去做了
7.2 本期不做
•	社区 / 评论 / 关注体系
•	长周期陪伴与人格养成
•	多轮复杂对话
•	高自由度 AI 图像生成
•	复杂账号体系
•	心理治疗与诊断相关功能
________________________________________
八、典型使用场景
场景一：求职卡住
用户输入：“我又不想投简历了。”
系统输出：状态命名 + 嘴替文案 + 贴纸 + 荒诞动机 + “只改简历标题”的微动作。
场景二：学习摆住
用户输入：“今天一天都没学进去。”
系统输出：将“我废了”转译为“当前处于高阻力低启动状态”，先用温和荒诞感接住，再建议“只打开文档写标题”。
场景三：想发疯但又不想完全烂掉
用户输入：“我现在只想发疯。”
系统输出：允许情绪表达，同时根据语气强度控制避免过量玩梗，并给出一个不刺激、能落地的小动作。
________________________________________
九、输出结构标准
每次结果统一为五段，控制在用户 15 秒内可扫完。
1. 状态命名
你现在是什么状态。
2. 抽象嘴替
一句符合年轻人语境、可截图传播的话。
3. 贴纸文案
一句更短、更适合视觉卡片的表达。
4. 荒诞动机
用同一套语气体系，把用户从情绪表达平滑带向行动，不允许直接跳成说教口吻。
5. 最小行动
一个可以立刻执行的动作，默认 3 分钟内完成。
示例：
用户输入：我又不想投简历了。
状态命名：你现在不是单纯懒，而是在高羞耻任务面前开启了回避模式。
抽象嘴替：某人类已在资本主义副本门口礼貌徘徊 27 分钟。
贴纸文案：高雅求职中，精神略乱。
荒诞动机：既然资本主义的羊毛不薅白不薅，现在改一行标题，也算对用人单位进行一次赛博碰瓷。
最小行动：现在只改简历标题，不许打开招聘软件。
贴纸文案约束： - 绝对不超过 25 个汉字 - 优先单句，不使用多段结构 - 避免连续标点、长括号和复杂符号 - 生成失败时允许自动降级为短版文案
________________________________________
十、产品原则
•	不说教，不训人
•	不强行励志，不假装热血
•	不乱用热梗，不硬凹年轻语气
•	先命中，再好笑，再推动行动
•	所有输出最终都要收束到一个现实动作
•	微行动必须通过“荒诞动机”过桥，避免语气断裂
•	抽象强度必须可控，不是所有场景都适合高浓度发疯
________________________________________
十一、AI 方案
MVP 采用轻量 pipeline，而不是直接一个 prompt 端到端生成。
Step 1：状态理解
模型输出结构化状态标签与语气强度建议。
Step 2：语境检索
从人工策展的小型语料库中召回匹配风格。
Step 3：结果生成
基于状态标签、语气强度和风格模板生成五件套。
Step 4：安全过滤
对过度负面、自伤、攻击性表达降级处理，同时对抽象强度做二次校正。
Step 5：贴纸渲染
将贴纸文案渲染成图片卡片，并执行长度与排版约束。
这样做的原因是：直接让模型随机生成，容易失控、过时、命中率低，也不利于后续调优。
________________________________________
十二、技术方案
前端： Next.js + TypeScript + Tailwind + shadcn/ui
后端： Next.js Route Handlers
数据库： Supabase / PostgreSQL
向量检索： pgvector
模型： OpenAI API（生成 + embedding）
贴纸渲染： HTML/CSS/Canvas 或 Satori + Resvg
分析埋点： PostHog
部署： Vercel
前端约束补充： - 贴纸卡片需支持文字自适应缩放 - 需支持自动换行与超长文案降级模板 - 长度超限时优先触发后端重生成，其次使用前端兜底裁切策略
技术方案遵循一个原则：够用、可跑、可演示，不追求重基础设施。
________________________________________
十三、数据策略
前期不做大规模抓取，采用“种子样本 + AI 扩量 + 人工审核”的方式完成语料冷启动，而不是纯手工逐条编写。
执行方式： - 先人工编写少量高质量 seed samples，明确风格边界 - 再使用高阶大模型配合 few-shot prompt 批量生成初始语料 - 最后由人工完成筛选、去重、微调和结构化标注
初始规模建议： 200–500 条。
覆盖场景： 求职、学习、工作、关系、日常低电量。
语料字段： 场景标签、情绪标签、抽象程度、是否适合贴纸、新鲜度、风险等级、适用语气强度。
目标不是“收录最多的梗”，而是建立一套足够新鲜、足够可控的表达底层。
________________________________________
十四、风险与边界
主要风险有四类：
1. 娱乐化过度。
把严肃情绪问题做成纯玩梗内容。
2. 语境失真。
模型说出来的话像品牌装年轻，不像真实用户表达。
3. 负面失控。
生成过度自贬、攻击或高风险内容。
4. 抽象失配。
用户只是低电量和疲惫，系统却输出过量玩梗，导致不适配。
处理原则： - 高风险输入不玩梗，转为稳定表达 - 不输出伤害性、羞辱性内容 - 不做诊断、治疗、疗效承诺 - 抽象强度由状态识别与安全过滤共同控制
________________________________________
十五、核心指标
MVP 主要看三类指标。
体验指标： - 被懂感评分 - 文案命中感评分 - 微行动可执行感评分 - 语气自然度评分
行为指标： - 结果复制率 - 贴纸保存率 - 重新生成率 - 微行动点击率 - 微行动完成自报率
求职展示指标： - 可稳定演示完整闭环 - 能清晰讲出 AI 设计与产品取舍 - 能形成简历项目与面试案例
________________________________________
十六、验收标准
MVP 达标的最低标准： - 用户可在 30 秒内完成一次完整使用 - 输出结果风格稳定，且结构完整 - 贴纸可以成功保存或复制，且主流场景下不出现明显排版崩坏 - 至少能支撑 3 个典型场景演示 - 项目可以被完整讲述为一个 AI PM 案例
________________________________________
十七、版本规划
Phase 1：求职展示版
先完成最小闭环，确保可用、可讲、可跑。
Phase 2：体验增强版
加入风格切换、历史记录和更细的反馈。
Phase 3：内容产品化版
再考虑每日贴纸、主题模板和更多内容运营能力。
________________________________________
十八、项目价值
对于求职 AI PM，这个项目的核心价值不在于做大，而在于它能完整展示以下能力： - 发现新型用户表达需求 - 将模糊想法收敛成清晰 MVP - 设计 AI 生成链路而不是简单接 API - 用内容产物提高体验完成度 - 用基础指标验证产品闭环
这应该被定位为一个高完成度的小项目，而不是一个尚未完工的大梦想。
________________________________________
十九、MVP 开发任务拆解
本拆解以“先跑通闭环，再优化质感”为原则，适合直接作为 Trae 内的开发任务清单使用。默认目标不是做完整产品，而是优先完成一个可演示、可截图、可讲述的最小闭环。
19.1 开发总原则
•	先做单链路闭环，再补边角功能
•	先保证结果可用，再追求风格惊艳
•	先做静态模板和规则兜底，再接 AI 动态生成
•	先做主路径，不在历史记录、账号、分享链路上分心
________________________________________
19.2 按优先级拆解任务
P0｜必须先完成的核心闭环
这一层的目标只有一个：让用户输入一句话后，系统能稳定返回一套完整的五件套结果，并且贴纸可以正常显示。只要这一层没打通，后面所有风格、语料和美化都没有意义。
第一组任务是项目初始化。需要创建 Next.js + TypeScript 项目，接入 Tailwind 和 shadcn/ui，配置环境变量，并确保本地开发环境能稳定启动。完成后，你应该至少能看到一个可访问的基础页面。
第二组任务是页面骨架。需要先搭建首页、结果区和贴纸区的静态布局。这里的目标不是好看，而是让信息架构站住：输入区、结果卡片区、贴纸预览区、反馈区，各自有明确位置。
第三组任务是输入模块。需要完成输入框、提交按钮、示例语句入口和加载态。做到这一步时，用户已经可以输入一句困境描述并触发一次生成流程。
第四组任务是输出结构。先不要接 AI，先用 mock 数据把五件套完整展示出来。也就是说，页面上先固定显示“状态命名、抽象嘴替、贴纸文案、荒诞动机、最小行动”五张结果卡片。这样可以先验证产品结构是不是顺的。
第五组任务是 AI 接口层。建立 /api/generate 路由，先打通最基础的模型调用，返回结构化 JSON。此时最重要的不是文案有多精彩，而是字段要稳定。
第六组任务是 Prompt v1。明确五个核心字段的输出要求，并增加 tone_level 字段。Prompt 的重点是结构稳定、字段不缺失、贴纸文案长度受控。
第七组任务是状态识别。需要在生成前增加状态标签和语气强度识别，至少能区分拖延、羞耻、疲惫、轻度发疯等几类状态，并映射到 L1、L2、L3 三挡语气模式。
第八组任务是微行动桥接。把“荒诞动机”正式写进生成链路，确保最小行动不是突然冒出来的命令，而是经过同一套语境自然过桥。
第九组任务是贴纸渲染 v1。先用一套固定模板，把贴纸文案渲染成可预览卡片。先不做多模板，也不追求精致视觉，只需要它像一个完成度不错的内容卡片。
第十组任务是贴纸约束。给贴纸文案加上长度限制和前端自适应缩放逻辑，避免溢出、压缩过小或直接炸版。这里需要后端和前端一起兜底。
第十一组任务是错误兜底。处理生成失败、字段缺失、文案超长、模型超时等异常情况。目标是不让页面在异常情况下直接报废。
第十二组任务是基础埋点。至少记录提交、重新生成、复制文案、保存贴纸、点击微行动等关键事件。对求职展示来说，这一步不是为了运营，而是为了证明你有验证意识。
P0 验收标准很简单：用户输入一句话后，可以稳定看到一套完整结果；贴纸能正常显示；五件套字段齐全；异常情况下页面不会崩掉。
________________________________________
P1｜完成度与风格稳定层
这一层的目标是让产品从“能跑”变成“像作品”，并让输出风格开始稳定下来。
第一组任务是语料库 v1。这里不要求你一开始就做满 500 条，而是先整理一批足够覆盖主要场景的 seed samples，然后用 AI 扩量生成，再人工审核和标注。前期只要能覆盖求职、学习、低电量日常三类场景就够了。
第二组任务是检索链路。把语料接入 embedding 和简单语义召回，让模型生成前先参考匹配风格。这样做的目标不是让输出更复杂，而是让同类输入下的结果更稳。
第三组任务是风格控制。把 L1、L2、L3 三挡强度真正落到 Prompt 和检索逻辑里。你需要明确每一挡允许什么语气，不允许什么语气，高风险和低电量场景默认下调到 L1。
第四组任务是反馈模块。把“像不像我想说的”“有没有帮我动起来”“我去做了”等按钮做出来，并把反馈结果存起来。这样你后面就能讨论哪些输出真的命中，哪些只是看起来有趣。
第五组任务是贴纸模板扩展。从一套模板扩到 5–10 套模板，但这里仍然不应该追求风格过多，而应该优先保证统一感和稳定性。
第六组任务是安全过滤。对高风险、自伤、攻击性输入进行降级，不输出高浓度玩梗内容。这里是为了给项目设边界，也方便你在面试里说明安全思考。
P1 验收标准是：同类输入下结果风格更一致；L1、L2、L3 差异可感知；用户能点反馈；高风险输入会自动降级；贴纸模板不止一套但仍然统一。
________________________________________
P2｜可选增强层
这一层不是 MVP 必需项，只在前两层做完且状态稳定后再考虑。
第一组任务是历史记录。保存最近几次生成结果，方便用户回看，也方便你自己调试和展示。
第二组任务是风格切换。允许用户手动选择“稳一点 / 抽一点 / 发疯一点”。这一功能会增强可玩性，但不是第一阶段必须有。
第三组任务是结果分享优化。包括优化截图区域、复制文案体验、贴纸导出细节等。这些都会提升传播感，但不应抢在闭环之前做。
P2 的判断标准不是“做不做都行”，而是“只有在闭环和稳定性已经达标后才值得做”。
________________________________________
19.3 推荐开发顺序
实际开发时，不建议平均铺开，而要按闭环顺序推进。
第一阶段是“假数据闭环”。目标是在完全不接 AI 的情况下，先把前端结构跑通。你需要完成项目初始化、页面骨架、输入模块、输出结构和贴纸渲染 v1。完成后，用户输入一句话，哪怕看到的是 mock 数据，也应该已经能体验到一个完整产品的形状。
第二阶段是“真实生成闭环”。目标是从假产品变成真产品。这里需要完成 AI 接口层、Prompt v1、状态识别、微行动桥接和错误兜底。完成后，用户真实输入一段文字，系统能稳定返回完整五件套，且“荒诞动机”能自然把内容桥接到动作。
第三阶段是“完成度修正”。目标是让产品看起来不再像工程 demo，而像作品。这一阶段重点做贴纸约束、基础埋点、反馈模块和贴纸模板扩展。完成后，产品已经具备录屏演示和写进简历的条件。
第四阶段是“风格稳定化”。目标是减少输出忽高忽低的问题。这里做语料库 v1、检索链路、风格控制和安全过滤。完成后，同类输入下的结果会更一致，不同强度模式之间也会更有层次。

19.4 模块级详细拆解
模块一：前端页面与交互骨架
这一模块负责把产品的基本形状搭出来。需要做首页 Hero 区、输入框、结果卡片区、贴纸预览区、反馈按钮区，以及加载态和错误态。开发时先用固定假数据把 UI 跑顺，不要一开始就绑死在 AI 输出上。结果区建议用卡片流，而不是长聊天流，因为卡片流更适合截图和求职演示。验收标准是一屏内能看清输入和核心结果，结构不需要解释也能理解。
模块二：生成接口与 Prompt 设计
这一模块负责让系统能返回结构化结果。需要建立 /api/generate 接口，约束 JSON 结构，明确五个输出字段：status_label、voice_line、sticker_text、absurd_motivation、micro_action，同时加入 tone_level。这里的核心不是追求句子惊艳，而是先追求字段稳定和边界清晰。Prompt 里要明确每个字段的字数要求、语气要求和禁止事项，尤其要把 sticker_text 单独限制长度，避免把排版问题完全丢给前端。验收标准是多轮测试下字段不乱、贴纸文案大体满足长度约束、最小行动足够具体。
模块三：状态识别与语气强度控制
这一模块决定产品“疯得对不对味”。需要定义状态标签集和 tone_level 三挡，并写清不同强度下的语言边界。高风险或低电量输入应该默认往低强度模式降。这个字段可以不暴露给用户，但后端一定要有。验收标准是：求职拖延、轻度发疯、低电量三类输入能明显得到不同强度的输出，不会出现用户只是很累，系统却突然高度玩梗的失配。
模块四：语料库与检索链路
这一模块负责提高风格稳定性。你需要先人工写 20–30 条高质量 seed samples，再用 AI 批量扩量，之后人工去重、删差、打标签，保存为结构化数据。前期不要追求大而全，只要先覆盖求职、学习、低电量三类场景即可。每条语料至少要带场景标签、抽象程度、适用强度等信息。验收标准是接入检索后，文案风格比纯生成更稳定、更像同一产品说出来的话。
模块五：贴纸生成与视觉约束
这一模块负责把“结果”变成“内容产物”。先做一套固定模板即可，然后动态填充 sticker_text。同时需要增加自动缩放、自动换行和超长降级策略，并提供保存图片和复制文案的能力。这里的重点永远是排版安全和截图感，而不是炫设计。验收标准是贴纸预览不炸版，用户可以顺利保存或复制，并且贴纸确实有一点“愿意发给朋友看”的感觉。
模块六：反馈、埋点与演示准备
这一模块直接关系到你后面的求职表达。需要埋点 submit、regenerate、copy、save、action_click、feedback_click 等关键事件，存储简单反馈数据，并准备 3–5 条高质量 demo 输入。这里的目标不是做运营后台，而是让你在讲项目时能说出“我定义了哪些验证指标、如何观察用户反应”。验收标准是关键行为都能被记录，产品打开后可以顺畅演示至少 3 个不同场景。

19.5 建议的 Trae 任务列表写法
如果你要直接丢进 Trae，可以按 Epic 拆。
Epic 1 是基础产品骨架。包括初始化 Next.js 项目与 UI 组件，完成首页输入区与结果区布局，完成五件套卡片组件，以及完成贴纸预览组件。
Epic 2 是 AI 生成链路。包括新建 generate API，设计五件套 Prompt v1，增加状态识别与 tone_level 输出，以及接入错误兜底与 fallback。
Epic 3 是贴纸与视觉稳定性。包括实现贴纸模板渲染，增加贴纸文字长度约束，实现文字自适应缩放与超长降级，并完成保存/复制能力。
Epic 4 是语料与风格控制。包括整理 seed samples，用 AI 批量扩量语料，完成语料标注与存储，并接入检索逻辑与强度控制。
Epic 5 是反馈与演示。包括完成反馈按钮与数据写入，接入基础埋点，并准备 demo case 与录屏脚本。

19.6 最小上线定义
当以下条件全部满足时，可以视为 MVP 已经达到“可以暂停继续扩功能，转去包装求职材料”的状态。
第一，用户输入一句困境后，可以稳定得到完整五件套结果。第二，结果中包含自然的“荒诞动机”桥接，不会突兀说教。第三，贴纸能正常显示、保存、复制，且主流情况下不崩版。第四，产品至少支持 3 个典型场景的稳定演示。第五，你可以用 3–5 分钟讲清楚用户问题、产品方案、AI 链路、关键取舍和验证指标。
到了这一步，这个项目就已经具备简历展示价值。

二十、3 天冲刺计划
这份冲刺计划不是平均分配工作量，而是围绕一个明确目标来设计：第 3 天结束时，你必须拥有一个可运行、可录屏、可讲述的 MVP。因此三天的安排遵循一个顺序：第一天先把产品“形状”搭出来，第二天把“AI 闭环”接通，第三天把“完成度和求职展示价值”补齐。
Day 1｜把产品形状搭出来
第一天的目标不是追求智能，而是先让这个项目从 0 变成一个“看起来已经是产品”的东西。你要做的是前端骨架、静态结果和基础交互，让页面本身先成立。
第一项任务是完成项目初始化。包括创建 Next.js + TypeScript 项目，接入 Tailwind 和 shadcn/ui，配置 .env.local，把页面基本框架跑起来。做到这里的标准是：本地项目能正常启动，你有一个干净的首页。
第二项任务是完成页面信息架构。把首页拆成输入区、结果区、贴纸预览区和反馈区。这里不需要美术级精修，但要有清晰层次。你现在做的是产品原型，不是登录页练手项目，所以重心是让页面结构一眼能懂。
第三项任务是做输入模块。包括文本输入框、示例语句、一键生成按钮、加载态和最基础的交互反馈。到这一步时，用户应该已经可以输入一句“我现在卡住了什么”，并触发一次完整流程。
第四项任务是做五件套静态卡片。先不要接模型，先用 mock 数据把“状态命名、抽象嘴替、贴纸文案、荒诞动机、最小行动”完整展示出来。这样你可以尽早验证一个关键问题：这个产品的输出结构是不是顺的，信息密度会不会过高，用户扫一眼能不能明白。
第五项任务是做贴纸预览 v1。先只用一套固定模板，把静态贴纸文案渲染成卡片。不要一上来追求多模板，也先不用管导出图片。第一天只要保证贴纸在页面里看着像样，不炸版，就已经够了。
第一天结束时的验收标准只有四个。第一，页面能正常打开。第二，用户可以输入一句话并看到完整五件套。第三，贴纸区域已经存在且结构稳定。第四，整个产品即使还没接 AI，也已经像一个真实产品，而不是一堆零散组件。
如果第一天提前完成，不要急着做复杂功能，优先补三个东西：页面文案统一、卡片层级优化、加载态更自然。这些小地方会极大影响你的“完成度观感”。
________________________________________
Day 2｜把 AI 闭环接通
第二天的目标是把“会动的壳子”变成“真的会生成内容的产品”。这一天最重要的不是输出多惊艳，而是让接口、字段和生成逻辑稳定下来。
第一项任务是新建 /api/generate 接口，接通 OpenAI API，返回结构化 JSON。这里的首要目标是字段稳定，而不是语言华丽。你要确保接口稳定返回这几个字段：status_label、voice_line、sticker_text、absurd_motivation、micro_action、tone_level。
第二项任务是写 Prompt v1。你需要在 Prompt 里把五件套的职责写清楚，把贴纸文案长度写死，把“荒诞动机”定义为桥接层，而不是额外装饰。Prompt 里还要写清 tone_level 三挡的语气边界，避免所有输入都输出成一个腔调。
第三项任务是接入状态识别与语气强度控制。你不需要单独训练分类器，但至少要让模型先判断当前输入更接近拖延、羞耻、疲惫还是轻度发疯，并输出一个 L1/L2/L3 的强度建议。这样你这个产品才不只是“写段子”，而是真的有控制逻辑。
第四项任务是把 AI 输出渲染回前端结果卡片，并替换掉 Day 1 的 mock 数据。做到这里时，整个产品才算真正打通。
第五项任务是完成“荒诞动机”桥接检查。你需要重点盯三组测试输入：求职拖延、学习摆住、想发疯。看最小行动前的过渡是不是顺。只要这里一生硬，整个产品的高级感就会掉下来。
第六项任务是做错误兜底。包括模型超时、字段缺失、返回格式不对、贴纸文案超长等情况。目标不是把所有异常都完美处理，而是保证产品在异常时不会直接崩掉，至少能给用户一个可理解的 fallback。
第二天结束时的验收标准也很明确。第一，用户真实输入一句困境后，系统能返回完整五件套。第二，不同输入下结果有一定差异，不是同一个模版反复套壳。第三，最小行动前的“荒诞动机”确实起到了桥接作用。第四，异常情况下页面不报废。
如果第二天还有余量，优先补的是贴纸文案长度约束和一次简单的人工回归测试，不要急着扩功能。
________________________________________
Day 3｜补完成度，准备求职展示
第三天的目标不是继续大扩张，而是把这个项目从“能跑”推进到“能演示、能录屏、能写进简历”。也就是说，这一天的关键词不是功能，而是稳定性、质感和表达力。
第一项任务是完成贴纸约束与视觉稳定性。把贴纸文案的长度硬约束真正接进生成逻辑，再补上前端的自适应缩放、自动换行和超长兜底。这里是你最容易在 demo 时翻车的地方，所以第三天必须收掉。
第二项任务是做最基础的反馈模块。把“像不像我想说的”“有没有帮我动起来”“我去做了”“重新生成”这些按钮做出来。哪怕先只记录到本地或简单写入数据库，也比没有强，因为这能让你在面试里讲“我有定义验证闭环”。
第三项任务是接基础埋点。至少把 submit、regenerate、copy、save、action_click 这些关键事件记录下来。对求职来说，埋点不是锦上添花，而是把你的项目从“做了个 demo”抬到“做了个会验证的产品”。
第四项任务是准备 3–5 条 demo case。不要临场想输入。你应该提前准备好三类最能体现产品价值的样例，比如“我又不想投简历了”“今天一天都没学进去”“我现在只想发疯”。并且每条都自己跑过，确认输出质量够稳。
第五项任务是做一轮手工验收。你需要自己把主链路走至少 10 次，重点检查四个点：字段会不会丢、贴纸会不会崩、语气会不会失配、微行动会不会太空。只有自己先踩过一轮坑，录屏演示时才不容易翻车。
第六项任务是整理求职展示素材。包括准备录屏、截几张关键页面图、整理一句项目定位、一段产品思路、一段 AI 链路说明，以及一段你为什么这样取舍的说明。第三天结束时，你不仅要有产品，还要有讲产品的话术。
第三天结束时的验收标准是：第一，产品主链路稳定可跑。第二，贴纸保存或复制不崩。第三，至少有 3 个典型场景可稳定演示。第四，你能在 3–5 分钟内讲清楚用户问题、方案、AI 设计、关键卡点和验证方式。
________________________________________
二十一、3 天冲刺的底线取舍
如果三天内时间真的非常紧，必须明确什么是绝对不能丢的，什么是可以延后的。
绝对不能丢的是这五样：页面骨架、五件套结构、AI 生成接口、荒诞动机桥接、贴纸预览。因为这五样共同构成了这个产品的最小辨识度。
可以延后的是这几样：多模板贴纸、语料检索完整接入、历史记录、复杂反馈系统、手动风格切换。这些都会加分，但不决定你有没有一个完整可讲的作品。
你要始终记住，这三天的任务不是“做完所有想法”，而是“做出一个已经足够像作品的小项目”。
________________________________________
二十二、3 天后应达到的状态
冲刺结束后，理想状态不是“这个产品已经完美”，而是你已经拥有以下四样东西：
第一，一个真实可运行的 Web MVP。第二，一套稳定的 demo 场景。第三，一份已经成型的 PRD 和开发拆解。第四，一段可以直接用于简历和面试的项目叙事。
如果这四样都拿到了，这个项目就已经完成了它对你当前求职最核心的使命。
